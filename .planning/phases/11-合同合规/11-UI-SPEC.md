| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FAFBFC` (`--bg-page`) | Page background |

## API Design (Frontend Contract API Module)

### contractApi.ts

```typescript
// File: frontend/src/api/contract.ts

import request from './request'

// === Types ===

export type ContractStatus =
  | 'draft'        // иҚүзЁҝ
  | 'pending_sign' // еҫ…зӯҫзҪІ
  | 'signed'       // е·Ізӯҫ
  | 'active'       // з”ҹж•Ҳдёӯ
  | 'terminated'  // е·Із»Ҳжӯў
  | 'expired'     // е·ІиҝҮжңҹ

export type ContractType = 'fixed_term' | 'indefinite' | 'intern'

export interface Contract {
  id: number
  employee_id: number
  employee_name: string
  type: ContractType
  type_label: string         // e.g. "еҠіеҠЁеҗҲеҗҢпјҲеӣәе®ҡжңҹйҷҗпјү"
  start_date: string         // YYYY-MM-DD
  end_date: string | null   // null for indefinite
  status: ContractStatus
  status_label: string
  expiry_days: number | null  // null for indefinite/no expiry
  created_at: string
  signed_at: string | null
  pdf_url: string | null
  signed_pdf_url: string | null
}

export interface ContractListResponse {
  list: Contract[]
  total: number
}

export interface CreateContractData {
  employee_id: number
  type: ContractType
  start_date: string
  end_date: string | null
}

export interface GeneratePdfResponse {
  pdf_url: string
}

export interface SendSignLinkResponse {
  message: string
  expires_at: string
}

export interface SignVerifyCodeData {
  contract_id: number
  phone: string
  code: string
}

export interface SignVerifyCodeResponse {
  token: string   // short-lived token for Step 3 confirm
}

export interface ConfirmSignData {
  token: string    // from Step 2 verify response
  contract_id: number
}

export interface ConfirmSignResponse {
  signed_pdf_url: string
}

// === API Methods ===

export const contractApi = {
  list: (employeeId: number) =>
    request.get<ContractListResponse>(`/employees/${employeeId}/contracts`)
      .then(r => r.data),

  create: (data: CreateContractData) =>
    request.post<Contract>('/contracts', data)
      .then(r => r.data),

  generatePdf: (contractId: number) =>
    request.post<GeneratePdfResponse>(`/contracts/${contractId}/generate-pdf`)
      .then(r => r.data),

  sendSignLink: (contractId: number) =>
    request.post<SendSignLinkResponse>(`/contracts/${contractId}/send-sign-link`)
      .then(r => r.data),

  // Employee H5 signing endpoints
  sendSignCode: (contractId: number, phone: string) =>
    request.post('/contracts/sign/send-code', { contract_id: contractId, phone }),

  verifySignCode: (data: SignVerifyCodeData) =>
    request.post<SignVerifyCodeResponse>('/contracts/sign/verify-code', data)
      .then(r => r.data),

  confirmSign: (data: ConfirmSignData) =>
    request.post<ConfirmSignResponse>('/contracts/sign/confirm', data)
      .then(r => r.data),

  terminate: (contractId: number) =>
    request.put(`/contracts/${contractId}/terminate`),

  getSignedPdf: (contractId: number) =>
    request.get<{ url: string }>(`/contracts/${contractId}/signed-pdf`)
      .then(r => r.data),
}
```

## H5 Signing Page Route

```
Route: GET /sign/:contractId?token={temporary_token}

The signing page is a standalone H5 route (no auth required):
- Path: /sign/:contractId (or /sign?contract_id=X&phone=Y)
- Layout: AppLayout (no sidebar, no nav вҖ” just logo header)
- State machine:
  1. input_phone вҶ’ 2. input_code вҶ’ 3. confirm_sign вҶ’ 4. success

URL contains contract_id and a short-lived phone verification token.
Token format: `{contract_id}:{phone_hash}:{expires_at_timestamp}`
```

## Implementation Flags (Non-Blocking)

These are notes for the planning/execution phase вҖ” not design contract issues.

**FLAG-1: H5 "иҝ”еӣһйҰ–йЎө" destination needs product decision**

The signing success state shows a "иҝ”еӣһйҰ–йЎө" button. After signing, the employee is not a registered user. Options:
- A: Remove the button (simplest вҖ” signing complete, employee closes the page)
- B: Link to the company's public landing page (if one exists)
- C: Show a static "зӯҫзҪІе®ҢжҲҗ" page with no navigation

Decision belongs to the product owner. Implementers: note this in the plan.

**FLAG-2: URL token vs API token вҖ” architect clarification needed**

The H5 Route section (lines 556вҖ“567) describes a URL query token (`/sign/:contractId?token=...`), but `contractApi.ts` signing flow uses a server-issued token returned in `verifySignCode` response body and passed to `confirmSign`. These two approaches need to be reconciled:
- URL token: stateless, self-contained link (simpler for employees вҖ” just open link)
- API token: server-controlled, more secure, can revoke

Recommended approach: URL token with `contract_id` + a short-lived HMAC-signed token in the URL. The `verifySignCode` вҶ’ `confirmSign` flow handles the SMS verification step, while the URL token proves the employee has the correct link. Architect to confirm in planning.
