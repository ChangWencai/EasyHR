## Color

Source: `variables.scss` CSS custom properties. Primary accent is `#7C3AED` (locked from Phase 10).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FAFBFC` (`--bg-page`) | Page background |
| Secondary (30%) | `#FFFFFF` (`--bg-surface`) | Cards, panels, drawer |
| Accent (10%) | `#7C3AED` (`--primary`) | **Reserved for:** primary CTA buttons, active tab indicator, wizard step completed, contract type selected card border |
| Destructive | `#EF4444` (`--danger`) | Destructive actions only: terminate contract, contract expiry past due |

**Accent reserved-for explicit list:**
- Primary action buttons (发送签署链接, 确认签署, 发起合同)
- Active/selected states (selected contract type radio card border)
- Step wizard completed step indicator
- Drawer tab active indicator (`--primary` bottom border)
- Toast success icon fill
- Contract type card selected state: `border: 2px solid var(--primary)`

**Contract expiry countdown color coding:**
| Condition | Color | CSS Variable | Usage |
|-----------|-------|-------------|-------|
| > 30 days | Success green | `--success` `#10B981` | Safe — far from expiry |
| 8–30 days | Warning amber | `--warning` `#F59E0B` | Attention needed |
| 1–7 days | Danger red (light) | `#EF4444` at 90% opacity | Urgent — expiring soon |
| ≤ 0 days | Danger red | `--danger` `#EF4444` | Expired / past due |

## API Design (Frontend Contract API Module)

### contractApi.ts

```typescript
// File: frontend/src/api/contract.ts

import request from './request'

// === Types ===

export type ContractStatus =
  | 'draft'        // 草稿
  | 'pending_sign' // 待签署
  | 'signed'       // 已签
  | 'active'       // 生效中
  | 'terminated'  // 已终止
  | 'expired'     // 已过期

export type ContractType = 'fixed_term' | 'indefinite' | 'intern'

export interface Contract {
  id: number
  employee_id: number
  employee_name: string
  type: ContractType
  type_label: string         // e.g. "劳动合同（固定期限）"
  start_date: string         // YYYY-MM-DD
```

## Implementation Flags (Non-Blocking)

These are notes for the planning/execution phase — not design contract issues.

**FLAG-1: H5 "返回首页" destination needs product decision**

The signing success state shows a "返回首页" button. After signing, the employee is not a registered user. Options:
- A: Remove the button (simplest — signing complete, employee closes the page)
- B: Link to the company's public landing page (if one exists)
- C: Show a static "签署完成" page with no navigation

Decision belongs to the product owner. Implementers: note this in the plan.

**FLAG-2: URL token vs API token — architect clarification needed**

The H5 Route section (lines 556–567) describes a URL query token (`/sign/:contractId?token=...`), but `contractApi.ts` signing flow uses a server-issued token returned in `verifySignCode` response body and passed to `confirmSign`. These two approaches need to be reconciled:
- URL token: stateless, self-contained link (simpler for employees — just open link)
- API token: server-controlled, more secure, can revoke

Recommended approach: URL token with `contract_id` + a short-lived HMAC-signed token in the URL. The `verifySignCode` → `confirmSign` flow handles the SMS verification step, while the URL token proves the employee has the correct link. Architect to confirm in planning.
