# Domain Pitfalls -- v1.3 新功能集成陷阱

**Domain:** HR SaaS (考勤管理/待办中心/薪资增强/社保增强/员工管理增强)
**Researched:** 2026-04-17
**Confidence:** HIGH (基于代码审查 + 领域知识 + 搜索验证)

---

## Critical Pitfalls

### Pitfall 1: 考勤排班跨天班次导致打卡归属日期错乱

**What goes wrong:**
夜班（如 22:00-06:00）跨越两个自然日。打卡记录归属日期判断错误，导致：(1) 当天出勤统计少了这个人，(2) 次日多了一个人，(3) 加班工时按错误日期的倍率计算。排班模式下三班倒更加复杂，"工作天"定义不清会导致连续上班36小时的异常无法被检测。

**Why it happens:**
开发者直觉用 `createdAt` 的自然日作为打卡归属日，忽略了班次定义的"工作天"概念。三班倒场景中，22:00上班的员工其"工作天"应该算前一天而非当天。

**How to avoid:**
- 班次模型必须包含 `workDateOffset` 字段：打卡时间 + offset = 归属工作日
- 考勤计算器接收 (employeeID, workDate) 而非 (employeeID, naturalDate)
- 打卡记录表设计：`punch_time`（实际打卡时间戳）+ `work_date`（归属工作日）+ `shift_id`（班次ID），三者分离
- PRD 要求3种打卡模式（固定时间/按排班/自由），共享同一个打卡归属算法

**Warning signs:**
- 打卡记录只有 `created_at` 没有独立的 `work_date` 字段
- 排班模型只有开始时间没有跨天标记
- 考勤统计接口按 `DATE(created_at)` 分组而非按 `work_date`

**Phase to address:** 考勤管理 Phase（打卡设置 + 今日打卡）

### Pitfall 2: 审批流状态机遗漏转换路径导致死锁

**What goes wrong:**
PRD 1.1 定义了 7 种请假类型（事假/病假/调休假/年假/婚假/产假/陪产假）+ 补卡/调班/出差/外出，共 11 种审批类型。如果状态机设计不全，会出现：(1) 审批被驳回后无法重新提交，(2) 已通过的审批被错误地再次操作，(3) 超时未审批的申请永远卡在 pending 状态。

**Why it happens:**
审批流比看起来复杂得多。只设计了"提交 -> 通过/驳回"的简单流程，没有考虑：撤回、超时自动处理、批量审批时的部分失败、与考勤统计的数据一致性（审批通过后要立即更新出勤数据）。

**How to avoid:**
- 为所有审批类型定义统一的状态机：`draft -> pending -> approved/rejected/cancelled/timeout`
- 每种状态转换记录操作人和时间（已有 ChangeHistory 模式可复用）
- 审批状态变更必须触发考勤数据刷新（pending 的请假不计入出勤，approved 的才计入）
- 超时机制：gocron 定时扫描 pending 超过 N 天的审批，根据策略自动处理或提醒
- 用状态转换表（transition table）而非 if-else 实现状态校验

**Warning signs:**
- 审批状态用 string 硬编码散落在 handler/service 各处
- 没有 `approval_transitions` 或 `approval_logs` 表
- 审批通过后没有回调机制通知考勤模块

**Phase to address:** 考勤管理 Phase（审批流 + 出勤月报）

### Pitfall 3: 调薪生效期历史追溯破坏已有工资数据

**What goes wrong:**
PRD 要求"调薪/普调"功能，调薪数据有"生效期限"。如果实现不当：(1) 调薪后历史月份的工资被覆盖（2月调薪影响了1月已确认的工资），(2) 同一个月多次调薪导致数据冲突，(3) 查询某月薪资时无法正确获取"当时生效"的薪资标准。

**Why it happens:**
现有 `salary_items` 表的 `effective_month` 字段是按月快照模式，但调薪逻辑可能直接 UPDATE 已有记录而非新建快照。历史追溯需要"查询 X 月 Y 日时该员工的薪资标准"，但系统只存了月度快照没有变更历史。

**How to avoid:**
- 调薪操作必须 INSERT 新记录，禁止 UPDATE 历月份的 salary_items
- 调薪模型设计：`salary_adjustments` 表，包含 `employee_id, effective_month, adjustment_type (individual/batch), before_values, after_values, created_by`
- 工资核算时，读取生效月份 <= 当月的最新一条调薪记录
- 普调（按部门批量调薪）与单人工调共用同一个模型，通过 `batch_id` 关联
- 已确认（confirmed/paid）的月份的 salary_items 禁止修改

**Warning signs:**
- 调薪直接 UPDATE salary_items 而非创建新记录
- 没有调薪历史表，只有当前值
- 工资核算读取薪资标准时没有 "取最新生效" 的排序逻辑
- 没有"已确认月份禁止修改"的保护

**Phase to address:** 薪资管理 Phase（调薪/普调）

### Pitfall 4: 病假系数计算不符合地方法规

**What goes wrong:**
PRD 明确要求"病假工资根据相关法规规则，按工龄挂钩计算"。中国各省市病假工资系数差异巨大（上海按工龄分5档：60%-100%，部分地区分7档：40%-100%），医疗期计算规则也很复杂（3-24个月，与实际工作年限和本单位工龄都相关）。如果硬编码一套规则：(1) 只适用部分地区，(2) 法规更新时需要改代码重新部署，(3) 病假工资低于最低工资80%的法定底线未校验。

**Why it happens:**
开发者可能只查了一套地区的规则就作为标准实现，忽略了中国的社保/劳工法规是省市级别的地方性法规。现有社保模块已有"按城市配置"的模式（`SocialInsurancePolicy`），但病假系数目前没有被纳入配置化范围。

**How to avoid:**
- 参照社保政策库模式，新建 `sick_leave_policies` 配置表：`city_id, effective_year, rules (JSONB: 工龄分档 + 对应系数 + 医疗期规则)`
- 病假工资计算引擎接收 (city_id, years_of_service, base_salary, sick_days) 参数
- 计算结果必须校验 `>= 当地最低工资标准 * 80%`，不满足时自动补齐并标记为"法定补齐"
- 医疗期累计：需要在员工维度跟踪当前计算周期内的累计病休天数
- 现有 `salary_items` 的"病假扣款"逻辑需要重构：不是简单扣款，而是 `基本工资 * (1 - 病假系数) * 病假天数 / 应出勤天数`

**Warning signs:**
- 病假系数硬编码在 Go 代码中
- 病假工资计算只考虑了天数没考虑工龄
- 没有医疗期累计跟踪
- 计算结果没有最低工资底线校验

**Phase to address:** 薪资管理 Phase（病假系数联动）+ 考勤管理 Phase（请假审批数据供给）

### Pitfall 5: 绩效系数影响薪资但缺少审计追踪

**What goes wrong:**
PRD 要求"绩效系数0%-100%"，直接影响绩效工资（`绩效工资 * 绩效系数`）。如果实现不当：(1) 系数变更没有历史记录，出了薪资纠纷无法追溯，(2) 某月设置系数后，下月默认值不明确（是保持上月值还是重置为100%），(3) 绩效系数设置后立即影响已核算但未确认的工资数据。

**Why it happens:**
绩效系数看起来简单（一个百分比数字），但它与工资核算的状态机紧密耦合。开发时容易只存一个"当前系数"字段，忽略了：(1) 系数是月度维度，不是全局维度，(2) 系数变更需要审计日志，(3) 系数变更时机与工资核算流程的交互。

**How to avoid:**
- 绩效系数表：`performance_coefficients (employee_id, year, month, coefficient, set_by, set_at)`
- 每月必须显式设置，没有默认继承（PRD说"需手动设置系数"）
- 工资核算时校验：该月是否已设置绩效系数，未设置则提示"请先设置绩效系数"
- 系数变更记录操作人、时间、变更前后值
- 已 confirmed/paid 的月份禁止修改系数

**Warning signs:**
- 绩效系数存在员工表上而非独立月度表
- 没有 `set_by` 和 `set_at` 审计字段
- 工资核算不校验系数是否已设置
- 系数修改没有检查工资状态

**Phase to address:** 薪资管理 Phase（绩效系数 + 工资核算联动）

### Pitfall 6: 社保欠缴状态流转逻辑缺陷

**What goes wrong:**
PRD 定义了复杂的状态流转：正常 -> 待缴 -> 欠缴 -> 已转出/未转出。实现时常见错误：(1) 状态转换条件不完整，导致某些情况下状态卡死，(2) 欠缴金额计算错误（上月欠缴 + 当月应缴），(3) 减员操作与欠缴状态冲突（有欠缴的员工不允许减员还是先补缴再减员），(4) 状态判定依赖时间条件（每月15日/25日为关键节点）但服务器时区处理不当。

**Why it happens:**
现有社保模块只有 3 个状态（pending/active/stopped），PRD 要求扩展为 5+ 个状态（正常/待缴/欠缴/已转出/未转出），且状态转换涉及多个时间节点（15日/25日）。开发者可能低估了状态机的复杂度，也低估了"离职联动社保减员"这个跨模块事务的一致性要求。

**How to avoid:**
- 状态机完整定义：`active -> pending_payment -> arrears -> transferred`，加上 `not_transferred`（离职未转出）
- 状态转换全部通过 Service 层方法，禁止直接 UPDATE status 字段
- 时间敏感的状态判定用定时任务（gocron）而非实时计算：每日凌晨刷新一次状态
- 减员前校验：有欠缴的记录提示"存在欠缴，是否确认减员"
- 状态变更记录写入已有的 `ChangeHistory` 表
- 缴费渠道（自主/代理）影响状态流转路径，用策略模式处理

**Warning signs:**
- 社保状态判断用 `time.Now()` 直接比较而非定时任务
- 没有 `arrears` 和 `pending_payment` 的中间状态
- 减员操作不校验欠缴状态
- 状态转换没有 ChangeHistory 记录

**Phase to address:** 社保公积金 Phase（数据看板 + 增减员优化 + 缴费渠道）

### Pitfall 7: 离职流程多模块联动的事务一致性

**What goes wrong:**
员工离职触发联动：(1) 员工状态 -> resigned，(2) 社保创建停缴提醒，(3) 工资表剔除该员工，(4) 待办中心生成离职待办事项。如果其中某步失败：(1) 员工已标记离职但社保仍在参保中（继续产生费用），(2) 离职后工资表还包含该员工，(3) 待办遗漏导致交接事项未完成。

**Why it happens:**
现有 `OffboardingService.CompleteOffboarding` 只触发了 `siHandler.OnEmployeeResigned`，还没有与工资表、待办中心联动。更关键的是，`ApproveResign` 中更新员工状态和离职状态是在两步独立 UPDATE 中完成的，如果第二步失败，员工已标记离职但离职审批记录未更新。

**How to avoid:**
- 离职审批通过的操作必须在同一个数据库事务中：更新 Offboarding 状态 + 更新 Employee 状态
- 联动操作使用事件队列（asynq）而非同步调用：审批通过 -> 发布"员工离职事件" -> 各模块异步消费
- 每个模块的消费逻辑必须幂等（重复消费不产生副作用）
- 提供一个"离职联动状态检查"接口，确认所有联动操作都已完成

**Warning signs:**
- 离职审批的两个 UPDATE 不在同一个事务中
- 联动操作是同步的（社保停缴失败会阻塞离职审批）
- 没有事件消费失败的重试机制
- 无法查询某个离职流程的联动完成情况

**Phase to address:** 员工管理 Phase（办离职优化）+ 待办中心 Phase（事项联动）

### Pitfall 8: 待办中心事项聚合的实时性与一致性

**What goes wrong:**
待办中心需要聚合来自多个模块的事项：合同到期提醒、社保缴费提醒、个税申报待办、审批待处理、调薪生效等。实现时常见错误：(1) 实时聚合查询太慢（跨5+张表 JOIN），(2) 缓存的待办数据与实际状态不同步（审批已处理但待办列表还显示），(3) 限时任务的时间计算依赖服务器时间而非用户时区。

**Why it happens:**
待办中心是典型的"读多写多"聚合视图。开发者可能：(1) 每次查询都实时聚合，导致慢查询，(2) 或者用缓存但没有失效机制，(3) 限时任务的"剩余时间"在前端用 `new Date()` 计算但服务器端存储的是 UTC 时间。

**How to avoid:**
- 待办数据采用"事件驱动写入"模式：各模块状态变更时，主动写入/更新 `todo_items` 表
- `todo_items` 表设计：`id, org_id, source_module, source_id, title, status, due_date, priority, created_at`
- 各模块通过统一的 `TodoService.CreateOrUpdateTodo()` 接口操作待办
- 完成率环形图直接查 `todo_items` 的 status 分布，不需要实时聚合
- 限时任务的截止时间用 CST（东八区）存储和计算

**Warning signs:**
- 待办列表通过跨表 UNION ALL 实时聚合
- 没有 `todo_items` 物化表
- 审批通过后待办列表没更新
- 限时任务用前端时间而非服务器时间

**Phase to address:** 待办中心 Phase（事项汇总 + 限时任务 + 完成率）

### Pitfall 9: 组织架构树性能在数据增长后急剧下降

**What goes wrong:**
PRD 要求"组织架构可视化"。如果用简单的 `parent_id` 邻接表 + 递归查询实现：(1) 查询子树需要递归 CTE，层级深时性能差，(2) 统计部门人数需要反复 JOIN，(3) 可视化渲染时一次性加载整棵树，数据量大时前端卡顿。

**Why it happens:**
小微企业（10-50人）的组织架构通常只有 2-3 层，看起来递归查询完全够用。但问题是：(1) 这个架构设计一旦固定，后续扩容到几百人会出问题，(2) 可视化渲染时即使后端查询快，前端渲染100+节点的树形图也可能卡顿。

**How to avoid:**
- 小微企业阶段用邻接表 + PostgreSQL 递归 CTE 完全够用（50人规模查询 < 5ms）
- 但代码层面抽象 Repository 接口，后续可切换为闭包表方案
- 部门人数用冗余字段 `employee_count` + 触发器/应用层维护，避免每次 JOIN 统计
- 前端可视化懒加载：初始只加载前 2 层，点击展开时再请求子节点
- 当前阶段不引入闭包表的复杂度，但模型设计要预留 `path` 或 `level` 字段

**Warning signs:**
- 部门人数统计通过 `COUNT + GROUP BY` 实时查询
- 组织架构 API 一次返回所有层级所有节点
- 没有 `level` 或 `path` 辅助字段
- 前端树组件一次性渲染全部节点

**Phase to address:** 员工管理 Phase（组织架构可视化）

### Pitfall 10: 工资核算公式变更后历史数据被覆盖

**What goes wrong:**
v1.3 新增了病假系数、绩效系数、加班费等复杂计算逻辑。如果直接修改现有的 `CalculatePayroll` 逻辑：(1) 新逻辑会影响历史月份的工资数据（重新打开历史月份发现数字变了），(2) 上线切换期间，同一月可能存在新老两种算法的工资数据。

**Why it happens:**
现有工资核算的 `PayrollItem` 使用快照模式存储（每次核算时写入明细），这是正确的设计。但 PRD 新增的加班费（1.5x/2x/3x）、病假系数等计算逻辑比原来复杂得多，直接修改 `calculatePayroll()` 函数而不做版本隔离。

**How to avoid:**
- 历史月份的工资数据不可变：`confirmed/paid` 状态的 PayrollRecord 禁止重新核算
- 新增的薪资项（加班费、绩效系数调整）通过新的 PayrollItem 类型添加，不修改已有项的计算逻辑
- 工资核算函数按月份参数走不同的计算分支：v1.3上线月之前用旧逻辑，之后用新逻辑
- 或者更简单：核算函数始终用最新逻辑，但只允许核算当前月和未来月，历史月份数据只读

**Warning signs:**
- `CalculatePayroll` 函数修改后没有按月份保护历史数据
- 没有"已确认月份禁止修改"的强制校验
- 上线时没有数据迁移策略

**Phase to address:** 薪资管理 Phase（薪资增强 + 工资核算联动）

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| 审批流硬编码 7 种请假类型 | 快速交付 | 新增类型需改代码+部署 | V1.3 MVP 可接受，V2.0 必须配置化 |
| 病假系数按一套地区规则硬编码 | 跳过政策库设计 | 外地客户无法使用 | 仅上线初期单城市可接受 |
| 组织架构用邻接表+递归CTE | 简单实现 | 规模扩大后查询慢 | 50人规模可接受，200人需重构 |
| 待办中心实时聚合查询 | 省去物化表维护 | 数据量增长后API超时 | 事项总量 < 1000条可接受 |
| 绩效系数存在员工表 | 减少表数量 | 无法月度追踪历史 | 不可接受，必须独立月度表 |
| 调薪直接UPDATE salary_items | 实现简单 | 历史追溯丢失 | 不可接受，必须INSERT新记录 |

## Integration Gotchas

| Integration Point | Common Mistake | Correct Approach |
|-------------------|----------------|------------------|
| 审批通过 -> 考勤数据更新 | 同步更新导致审批响应慢 | 异步事件（asynq），但保证最终一致性 |
| 请假审批 -> 薪资扣款 | 审批状态和薪资计算的时间差 | 薪资核算时重新查询审批终态，不信任中间状态 |
| 离职审批 -> 社保减员 | 社保模块调用失败阻塞离职 | 事件驱动 + 重试 + 人工补单 |
| 调薪生效 -> 工资核算 | 调薪生效日与核算日冲突 | 核算前校验当月是否有未生效的调薪记录 |
| 社保增员 -> 员工入职 | 入职流程中社保增员可选 | 入职完成时提示"未配置社保"，不阻断入职 |
| 待办中心 <- 各模块状态变更 | 忘记某模块的状态变更通知 | 用事件总线模式统一收口 |
| 个税上传 -> 工资表更新 | Excel解析失败导致个税数据丢失 | 解析失败不写入，保留原有个税值并报错 |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| 出勤月报按日聚合查询 | 月报页面加载 > 3s | 预计算月度汇总表，每日定时刷新 | 员工数 * 30天 > 1500条记录 |
| 待办中心跨5表UNION ALL | 待办列表加载 > 1s | 物化 todo_items 表 + 事件驱动更新 | 事项总量 > 1000条 |
| 工资核算逐员工串行计算 | 一键核算 50人 > 10s | 并发核算（goroutine池）+ 批量写入 | 员工数 > 30 |
| 组织架构递归CTE | 部门树加载 > 500ms | 限制层级 + Redis缓存 + 邻接表加path字段 | 层级 > 5 或部门 > 50 |
| 审批列表按类型过滤 | 审批页分页查询 > 1s | (type, status, org_id) 联合索引 | 审批记录 > 5000条 |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| 审批操作只校验 org_id 不校验操作人角色 | 普通员工审批自己的请假 | 审批 API 必须校验操作人 >= ADMIN 角色 |
| 调薪金额没有审计日志 | 薪资纠纷无据可查 | 所有薪资变更记录操作人、时间、变更前后值 |
| 待办中心暴露其他人的待办事项 | 信息泄露 | 查询强制加 org_id + 角色/操作人过滤 |
| 考勤打卡位置数据未加密 | 员工隐私泄露 | 打卡位置用非精确坐标（精度到100米级） |
| 绩效系数接口没有频率限制 | 恶意批量修改 | 限制单次操作数量 + 操作间隔 |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| 审批列表显示所有类型混在一起 | 老板找不到重点 | PRD已要求Tab分类，默认显示待处理 |
| 考勤设置3种模式放在同一页面 | 老板不知道选哪个 | 引导式选择：先选模式，再配置细节 |
| 病假系数自动计算但没有解释 | 老板不理解数字怎么来的 | 显示计算公式 + 工龄信息 + "根据XX市规定" |
| 调薪只能改总额不能改明细 | 无法精确调薪 | PRD已要求分项调整（岗位工资/补贴/奖金/年终奖） |
| 出勤月报显示30天明细太密集 | 信息过载 | 默认显示汇总，点击展开某天明细 |
| 社保欠缴状态没有操作引导 | 老板看到欠缴不知道怎么办 | 状态旁显示"去缴费"按钮 |

## "Looks Done But Isn't" Checklist

- [ ] **考勤打卡:** 跨天班次打卡归属正确 -- 验证：创建 22:00-06:00 班次，测试打卡归属日期
- [ ] **审批流:** 驳回后可重新提交 -- 验证：提交请假 -> 驳回 -> 重新提交 -> 通过
- [ ] **审批流:** 审批状态变更触发考勤更新 -- 验证：请假审批通过后，出勤月报自动更新
- [ ] **调薪:** 历史月份工资不受影响 -- 验证：3月调薪后，查看2月工资数据未变化
- [ ] **调薪:** 普调按部门批量生效 -- 验证：部门3人同时调薪，每人独立记录生效
- [ ] **病假系数:** 最低工资底线校验 -- 验证：设置极低基本工资，病假工资不低于最低80%
- [ ] **绩效系数:** 未设置时阻断工资核算 -- 验证：未设系数直接核算，提示先设置
- [ ] **社保欠缴:** 15日/25日状态自动流转 -- 验证：定时任务手动触发，检查状态变更
- [ ] **社保减员:** 离职联动社保停缴 -- 验证：办离职后，社保列表出现停缴提醒
- [ ] **待办中心:** 审批完成后待办自动消失 -- 验证：处理一个审批，待办列表实时更新
- [ ] **限时任务:** 过期任务状态正确更新 -- 验证：超过截止日期的任务显示"超时"或"失效"
- [ ] **组织架构:** 部门人数实时准确 -- 验证：入职/离职后，部门人数立即更新
- [ ] **工资核算:** 加班费1.5x/2x/3x正确计算 -- 验证：工作日/双休日/节假日各一条加班记录
- [ ] **个税上传:** Excel解析失败不破坏现有数据 -- 验证：上传错误格式文件，工资表个税值不变

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| 考勤归属日期错误 | HIGH | 需要重新计算所有受影响月份的考勤数据 |
| 审批流死锁 | MEDIUM | 手动数据库修复状态 + 补充正确的转换路径 |
| 调薪覆盖历史数据 | CRITICAL | 如果没有备份则无法恢复，必须从备份恢复 |
| 病假系数违规 | MEDIUM | 修改配置 + 补发差额工资 |
| 绩效系数无审计 | LOW | 补充审计日志表，后续操作自动记录 |
| 社保状态流转错误 | MEDIUM | 停服修复状态 + 补缴流程 |
| 离职联动不一致 | HIGH | 逐个员工检查各模块状态并手动对齐 |
| 待办数据不同步 | LOW | 清空待办表 + 各模块重新写入 |
| 组织架构查询慢 | LOW | 加缓存或加 path 字段，无需停服 |
| 工资公式变更覆盖 | CRITICAL | 必须从备份恢复历史工资数据 |
