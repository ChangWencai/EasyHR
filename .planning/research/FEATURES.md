### 2.3 请假类型与薪资影响（关键业务规则）

| 假类型 | 薪资影响 | 计算规则 | 法律依据 |
|--------|---------|---------|---------|
| 事假 | **无薪** | 日工资=月基本工资/21.75，事假天数*日工资扣除 | 《劳动法》 |
| 病假 | **部分带薪** | 病假工资=基本工资*病假系数；系数与工龄和医疗期挂钩：6个月内按60%-100%，超6个月按40%-60%；不得低于当地最低工资80% | 《企业职工患病或非因工负伤医疗期规定》 |
| 调休假 | **带薪** | 以审批通过为准，不扣工资 | 企业内部制度 |
| 年假 | **带薪** | 工龄<10年=5天，10-20年=10天，>20年=15天；未休年假需支付300%日工资 | 《职工带薪年休假条例》 |
| 婚假 | **带薪** | 各省3-30天不等，按当地标准 | 各地《人口与计划生育条例》 |
| 产假 | **带薪** | 基础98天+各省奖励假(15-90天)；生育津贴由社保基金支付 | 《女职工劳动保护特别规定》 |
| 陪产假 | **带薪** | 各省15-30天不等 | 各地《人口与计划生育条例》 |

### 2.4 审批流状态机

```
员工提交申请
    |
    v
[pending] ──老板同意──> [approved] ──自动更新──> 假勤数据/考勤记录
    |                      |
    |                      v
    |                  联动薪资核算(病假扣款/加班费/事假扣款)
    |
    +──老板驳回──> [rejected]
    |
    +──员工撤回──> [cancelled]
    |
    +──终止──> [terminated]
```

**关键设计决策：**
- 企业版（EasyHR定位）不展示审批人和抄送人，老板直接审批
- 审批类型统一抽象：leave(请假)/makeup_card(补卡)/shift_change(调班)/business_trip(出差)/go_out(外出)
- 通过 approval_type + subtype 区分不同请假类型
- 审批通过后自动更新出勤统计，联动薪资核算

### 3.3 调薪数据模型设计要点

```
调薪记录模型(SalaryAdjustment):
- employee_id / department_id (单人调薪/部门普调)
- adjustment_type: individual / department
- 调整项: base_salary / performance / allowance / bonus / annual_bonus / deduction
- 调整前金额 / 调整后金额 / 调整比例
- effective_month (生效月份 YYYY-MM)
- status: pending / active / expired

生效逻辑:
- 当月薪资核算时，查询 effective_month <= 当月 的最新调薪记录
- 调薪记录叠加到基础薪资项上
- 多次调薪按时间取最新值
```

### 3.4 薪资核算增强算法

PRD 1.1 要求的薪资算法（比现有 calculator.go 大幅增强）：

| 算法 | 公式 | 输入来源 |
|------|------|---------|
| 基本工资 | 基本工资/应出勤 * 计薪天数 + 病假工资 | 考勤模块(应出勤/计薪天数) + 病假审批(病假天数/系数) |
| 绩效工资 | 绩效工资基数 * 绩效系数(0%-100%) | 绩效系数设置 |
| 加班费 | (基本工资/应出勤/8h) * 加班时长 * 系数(150%/200%/300%) | 考勤模块(加班审批记录) |
| 补贴 | 固定补贴(勾选每月固定) + 调薪补贴(生效期内) | 薪资方案 + 调薪记录 |
| 个税 | 从上传的个税表抓取，每月1日归零 | 个税上传模块 |

| 组织架构可视化 | 直观展示公司组织结构：部门->岗位->员工 | Med | 使用 ECharts Tree 或 vue3-tree-org 组件；数据来源于部门+岗位+员工关系 |
| 员工信息登记（可重复创建） | 同一员工可重新创建信息登记，以最新版为准 | Med | 通过手机号或身份证号匹配已有员工；覆盖更新而非新建 |

### 5.3 员工信息登记流程

```
管理员创建登记表
    |-- 填写必填项：姓名、部门、岗位、入职日期
    |-- 选择：自己填写 or 转发员工填写
    |
    +--[自己填写]--> 直接填写手机号/住址/身份证/银行卡等 --> 提交
    |
    +--[转发员工填写]--> 生成临时链接(Token)
                          |-- 短信/微信发送给员工
                          |-- 员工打开H5页面填写
                          |-- 提交后自动更新员工档案
```

### 5.4 Feature Dependencies

```
员工数据看板 ←--- employee 模块(聚合查询)
    |-- 在职人数(status=active/probation)
    |-- 新入职(hire_date 在当月)
    |-- 离职(status=resigned 且 resignation_date 在当月)
    |-- 离职率计算

组织架构可视化 ←--- employee 模块(position) + 部门数据
    |-- 需要部门层级关系（目前 Employee 只有 position 字段）
    |-- 可能需要新增 department 模型或从 position 推导

员工信息登记 ←--- employee + invitation 模型
    |-- 复用 Invitation 的 Token 机制
    |-- 增加信息登记专用的表单字段

办离职优化 ←--- offboarding + socialinsurance
    |-- 离职审批 -> 跳转社保减员(深度链接)
    |-- 减员完成后回调更新离职状态

花名册增强 ←--- employee + contract + salary 模块
    |-- 在职年限: now() - hire_date
    |-- 合同到期: contract.end_date - now()
    |-- 岗位薪资: 基本工资+绩效工资+固定补贴
```

## 六、Anti-Features（明确不做的功能）

| Anti-Feature | Why Requested | Why Problematic | What to Do Instead |
|--------------|---------------|-----------------|-------------------|
| GPS定位打卡 | 看似标准功能，竞品都有 | 需要移动端APP原生支持，H5无法获取可靠GPS；V1.3仅H5管理后台，无APP端 | V1.3做"老板端查看/管理"，打卡通过邀请链接H5点签或微信小程序 |
| 人脸识别打卡 | 高科技感，防作弊 | 需要对接人脸识别API（成本+隐私合规），H5端摄像头体验差 | 不做生物识别，V1.3用点签/拍照模式 |
| 复杂工作流引擎 | 审批流可自定义节点 | 10-50人企业不需要多级审批，过度工程化；Go生态工作流引擎成熟度不如Java | 固定单级审批（员工提交 -> 老板同意/驳回），不走工作流引擎 |
| 排班自动优化算法 | 智能排班是卖点 | 算法复杂度极高（NP-hard问题），V1.3用户规模不需要 | 手动排班（管理员分配），不搞智能排班 |
| 考勤硬件对接 | 指纹机/门禁卡/人脸终端 | 对接协议各异，售后成本高，超出SaaS产品范畴 | 仅支持软件打卡（H5点签/拍照） |
| 请假审批多级流转 | 大企业需要多级审批 | EasyHR定位小微企业，老板就是最终审批人 | 固定单级审批，未来如需要再扩展 |
| 实时考勤大屏 | 看起来很酷 | 10-50人企业不值得投入大屏，老板用手机看 | 今日打卡列表已满足需求 |

## 八、Feature Dependencies (Cross-Module)

### 核心依赖链

```
[考勤管理] ──必要依赖──> [薪资增强-考勤联动]
    | 出勤天数/请假天数/加班时长 是薪资核算的输入
    | 如果考勤模块不完成，薪资算法增强无法实现

[审批流] ──必要依赖──> [出勤月报]
    | 请假/补卡/出差/外出 审批数据参与出勤统计

[待办中心-限时任务] ──弱依赖──< [各业务模块]
    | 限时任务从各模块聚合数据，但不影响各模块核心功能
    | 可先建框架，逐步接入数据源

[社保增减员优化] ──联动──> [员工管理-办离职优化]
    | 离职审批通过后可跳转减员

[调薪记录] ──增强──> [薪资核算]
    | 调薪数据参与当月薪资计算

[绩效系数] ──增强──> [薪资核算]
    | 绩效系数影响绩效工资金额
```

### 实施顺序建议

```
Phase 1: 基础展示 + 待办中心
  - 员工/社保/薪资 数据看板（纯展示，无复杂业务逻辑）
  - 待办中心框架 + 限时任务引擎
  - 花名册增强
  --> 价值：老板打开系统立即获得有用信息

Phase 2: 考勤管理 + 审批流 + 薪资增强(基础)
  - 打卡设置(3种模式)
  - 审批流(请假/补卡/出差/外出/调班)
  - 今日打卡 + 出勤月报
  - 调薪/普调 + 绩效系数 + 薪资看板
  --> 价值：考勤和审批闭环，薪资基础增强

Phase 3: 数据联动 + 模块增强
  - 薪资算法增强(考勤联动)
  - 个税上传自动抓取
  - 社保增减员优化 + 缴费渠道 + 欠缴管理
  - 员工信息登记 + 办离职优化
  - 组织架构可视化
  - 待办中心完善(邀请协办/轮播图)
  --> 价值：全模块数据打通，一体化体验
```

## 十、Feature Complexity Summary

### 最高复杂度功能 (需要重点关注)

| Rank | Feature | Complexity Root Cause | Mitigation |
|------|---------|----------------------|------------|
| 1 | 排班管理 | 班次定义+排班周期+员工分配+特殊日期+调班审批，数据模型复杂 | V1.3先做最简版：固定班次+手动排班表 |
| 2 | 薪资算法增强(考勤联动) | 基本工资/加班费/病假工资计算依赖考勤数据，算法规则多 | 分步实现：先基本工资/事假扣款，再加病假/加班 |
| 3 | 审批流(请假7种类型) | 每种假类型有不同的薪资影响和天数计算规则 | 统一审批模型+类型配置，规则引擎化 |
| 4 | 个税上传自动抓取 | Excel解析+关键字匹配+容错处理，用户上传的表格格式不统一 | 先支持标准格式，非标准格式明确提示重新上传 |
| 5 | 限时任务引擎 | 7种任务的触发规则不同，超时/失效状态管理，定时任务调度 | 使用 gocron 统一调度，每种任务类型独立配置 |
