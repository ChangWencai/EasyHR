# 「易人事（EasyHR）」项目全面评估报告

> 评估日期：2026-04-10
> 评估基于：prd.md、tech-architecture.md、ui-ux.md、CLAUDE.md（STACK）

---

## 一、项目概述

**定位**：为小微企业/个体户（10-50人）打造的轻量化一站式人事管理APP，核心理念是「简单、好用、省时间」。

**文档体系**：
- prd.md — 产品需求文档（PRD），约11KB
- tech-architecture.md — 技术架构与详细设计，约62KB
- ui-ux.md — UI/UX设计原型说明，约21KB
- CLAUDE.md — 项目上下文配置（STACK、约束等）

## 二、项目亮点（做得好的部分）

### 2.1 产品定义清晰
- 目标用户画像精准：无专职HR的小微企业老板，特征描述到位
- 痛点分析准确：社保、个税、工资核算等确实是小老板的核心痛点
- 竞品差异化明确：钉钉/飞书以协同为主，人事功能非专门化
- 商业模式合理：V1.0免费积累用户 -> V2.0/V3.0增值变现

### 2.2 技术栈选型务实
- Go + Gin + GORM + PostgreSQL 是成熟且高性能的组合
- 模块化单体（Modular Monolith）架构决策正确，V1.0不需要微服务
- 各技术选型都附带了替代方案对比和选择理由，决策过程透明
- 单二进制 + Docker部署，运维成本极低（~570元/月）

### 2.3 数据设计专业
- 数据库设计非常详细，含30+张表的完整字段定义
- 安全设计到位：AES-256-GCM加密敏感字段 + SHA-256哈希索引（支持精确查找）
- 软删除 + 部分唯一索引避免冲突
- 审计追踪字段统一（created_by/updated_by等）
- 外键关系表清晰，含删除策略

### 2.4 UI/UX设计理念优秀
- 「3步完成核心操作」作为硬性约束贯穿始终
- 首页待办优先设计：老板打开APP第一时间知道要做什么
- 新手引导、空状态、错误状态等都有规范
- 财务模块独立为底部Tab，避免与人事功能混淆

### 2.5 安全设计全面
- JWT + 设备绑定 + 令牌轮换 + 黑名单
- API分级限流策略
- 请求签名防篡改（HMAC-SHA256）
- 个人信息保护法 + 劳动合同法合规方案
- 数据备份策略（全量/增量/归档三级）

### 2.6 迭代规划合理
- V1.0核心免费 -> V2.0增强 -> V3.0付费增值，节奏清晰
- 第三方依赖均设计降级方案，核心业务不依赖外部接口
- 技术债务管理表明确列出V1.0妥协和V2.0优化方向

## 三、存在的问题与风险

### 3.1 【严重】V1.0功能范围过大

**问题**：PRD中V1.0包含5大模块（员工管理、社保、工资、个税、财务记账），其中财务代理记账模块本身就是一个完整产品。

**影响**：
- 交付节奏写的是16-20周（4批 x 4-6周），这对小团队来说非常激进
- 财务模块（凭证、发票、报销、账簿、报表、会计期间、科目管理、税务计算）功能密度极高
- 「3步完成操作」的承诺在财务记账场景下很难兑现——凭证录入至少需要选科目、填金额、写摘要

**建议**：
- 将财务记账模块拆分到V1.5，V1.0聚焦「入职-社保-工资-个税」四大模块
- 或者财务模块V1.0仅做「费用报销 + 简易收支记录」，不做完整记账

### 3.2 【中等】前端开发量较大（已有缓解）

**现状**：项目原计划需要同时开发5个端，但 `miniprogram/` 目录已完成员工端微信小程序（8个页面已实现），H5管理后台 `frontend/` 也在推进中。

**当前已实现**：
- ✅ 微信小程序（员工端）：`miniprogram/` 完整实现 8 个页面，采用原生小程序 + WeUI
- ✅ H5管理后台：`frontend/` 使用 Vue 3 + Element Plus（进行中）
- ⏳ Android原生APP：待开发
- ⏳ iOS原生APP：待开发

**建议**：
- V1.0 优先上线 H5 + 小程序，覆盖 PC 和移动两个场景
- Android/iOS 原生 APP 延后至 V1.5，或考虑 React Native/Flutter 替代双端原生

### 3.3 【中等】社保政策库维护成本被低估

**问题**：V1.0计划自建30+城市的社保政策库，管理员手动更新。

**影响**：
- 中国有300+地级市，社保政策每年至少调整1次（基数调整），部分城市政策变动更频繁
- 30城市仅覆盖一线城市和部分二线城市，目标用户（小微企业）可能分布在任何城市
- 手动维护30城市 x 5险种 x 2比例 = 300+数据点的准确性很难保障
- 社保计算错误直接导致用户信任崩塌

**建议**：
- V1.0聚焦10个核心城市（用户最集中的），其他城市走「手动输入模式」
- 尽早评估第三方社保数据服务商（如51社保），将对接成本纳入V1.0预算
- 增加「社保计算结果人工确认」环节（已提到，但需确保是强制的）

### 3.4 【中等】技术栈文档存在不一致

**问题**：CLAUDE.md中的STACK.md与tech-architecture.md存在差异：
- STACK.md明确选GORM v1.31.1，但tech-architecture.md写的是「Ent / GORM」
- STACK.md明确选Gin v1.12.0，但tech-architecture.md写的是「Go 1.22+ + Gin/Fiber」
- STACK.md明确原生小程序不用Taro，tech-architecture.md写的是「原生小程序 / Taro 3」
- Go版本：STACK.md写1.23+，tech写1.22+
- PostgreSQL版本：STACK.md写16+，tech写15+

**影响**：开发团队参考不同文档会导致技术选型混乱。

**建议**：以STACK.md为最终决策，tech-architecture.md需要同步更新。

### 3.5 【中等】个税计算的业务复杂度

**问题**：个税自动计算看起来简单，但实际上非常复杂：
- 累计预扣法需要跨月累计数据
- 专项附加扣除的认定和变更
- 年终奖单独计税 vs 合并计税
- 不同城市的社保公积金政策影响税前扣除

**影响**：如果个税计算错误，用户面临税务风险。

**建议**：
- V1.0的个税计算结果需明确标注「仅供参考，最终以税务局核定为准」
- 增加个税计算的免责条款
- 考虑集成成熟的个税计算SDK而非自研

### 3.6 【低】缺少API接口的完整定义

**问题**：tech-architecture.md只给了API规范和几个示例URL，没有完整的接口列表。

**影响**：前后端协作时缺乏明确的接口契约。

**建议**：使用Swagger/OpenAPI生成完整的API文档，STACK.md中已选了swag。

### 3.7 【低】监控和运维方案偏弱

**问题**：tech-architecture.md提到了Prometheus + Grafana + Sentry，但V1.0的技术债务表写的是「基础日志」，V2.0才是完整APM。

**影响**：V1.0阶段如果出现问题，排查效率会很低。

**建议**：至少在V1.0接入Sentry（异常追踪）和基础的健康检查接口。

## 六、核心建议

1. **缩小V1.0范围**：砍掉或简化财务记账模块，聚焦「入职-社保-工资-个税」四大模块
2. **减少前端端数**：V1.0先做H5+小程序，原生APP延后
3. **统一技术文档**：以STACK.md为基准，更新tech-architecture.md
4. **社保数据优先级**：10城起步 + 手动输入兜底
5. **增加免责机制**：个税/社保计算结果需标注仅供参考
6. **补充API文档**：用swag生成OpenAPI文档，作为前后端契约

---

## 七、总结

总的来说，这是一个**方向正确、技术方案扎实**的项目，最大的风险是**V1.0功能范围过大导致交付延期**。建议做减法，先交付一个真正「简单好用」的MVP，再逐步迭代。
